## Prerequisites

- [Bun](https://bun.com/) — current stable release
- TypeScript `^5` (included as a peer dependency)

<Note>
  bun-api-kit is a Bun-native project. Node.js is not supported — the server uses `Bun.serve` directly.
</Note>

---

## Get up and running

<Steps>
  <Step title="Clone the repository">
    ```bash
    git clone https://github.com/pimatis/bun-api-kit
    cd bun-api-kit
    ```
  </Step>

  <Step title="Install dependencies">
    ```bash
    bun install
    ```

    This installs `rate-limiter-flexible` (rate limiting) and `zod` (input validation). No HTTP framework is installed — the server uses `Bun.serve` directly.
  </Step>

  <Step title="Configure environment">
    ```bash
    cp .env.example .env
    ```

    The `.env.example` file contains sensible production-oriented defaults:

    ```bash .env.example
    # Runtime mode
    NODE_ENV=production

    # Logging
    LOG_LEVEL=info

    # Request body limits in bytes
    MAX_BODY_BYTES=1048576

    # Comma-separated browser origins allowed by CORS in production
    CORS_ALLOWED_ORIGINS=https://api.example.com,https://app.example.com

    # Trusted reverse proxy IPs or IPv4 CIDR ranges
    TRUSTED_PROXY_RANGES=127.0.0.1/32,10.0.0.0/8
    ```

    For local development you can leave these at their defaults. See [Environment variables](/guides/environment-variables) for the full reference.
  </Step>

  <Step title="Start the server">
    ```bash
    bun run start
    ```

    This runs `bun --watch index.ts`, which starts the server on port 3000 and automatically reloads on file changes.

    You should see:

    ```
    [INFO] Listening on http://localhost:3000
    ```

    <Note>
      The server always starts on port **3000**, hardcoded in `index.ts`. Hot reload (`bun --watch`) is suitable for development. For production, run `bun index.ts` directly or use a process manager.
    </Note>
  </Step>

  <Step title="Send your first request">
    ```bash
    curl http://localhost:3000/api/v1/home
    ```

    Expected response:

    ```json
    { "success": true }
    ```

    Every endpoint returns a consistent `{ success, data? }` JSON envelope via the `apiSuccess` and `apiFailure` helpers. See [JSON responses](/concepts/json-responses) for details.
  </Step>

  <Step title="Check the health endpoint">
    ```bash
    curl http://localhost:3000/api/v1/health
    ```

    The health endpoint is designed for Kubernetes liveness/readiness probes and load balancer checks. It returns a minimal status payload:

    ```json
    {
      "success": true,
      "data": {
        "status": "healthy",
        "timestamp": "2024-01-15T10:30:00.000Z"
      }
    }
    ```
  </Step>
</Steps>

## What's happening on each request

When you hit `GET /api/v1/home`, the server runs through this pipeline:

1. **CORS preflight check** — OPTIONS requests are handled early and returned immediately
2. **Request context** — a unique `rid` is assigned (from `X-Request-ID`, `X-Correlation-ID`, or a generated UUID)
3. **Tracing** — a `traceId` and `spanId` are generated for distributed tracing
4. **Access log** — `[INFO] GET /api/v1/home client=127.0.0.1 traceId=...`
5. **Metrics** — request count and latency are recorded
6. **Route handler** — `home.ts` matches the path, applies body limits and rate limiting, returns a response
7. **Finalize** — security headers, `X-Request-ID`, and `X-Trace-ID` are merged onto the response
8. **Compression** — JSON responses larger than 1 KiB are compressed (gzip/br)

See the [Request pipeline](/concepts/request-pipeline) page for the full breakdown.
