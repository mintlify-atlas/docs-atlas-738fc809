---
title: "Environment variables"
description: "All supported environment variables, their defaults, and how to configure them for production."
---

bun-api-kit reads configuration entirely from environment variables. There are no required variables — the server starts with safe defaults out of the box. Set variables in a `.env` file for local development or via your deployment platform's secrets management for production.

## Setup

Copy `.env.example` to `.env` and edit the values for your environment:

```bash
cp .env.example .env
```

The `.env.example` file in the repository shows every supported variable with production-oriented values:

```bash
# Runtime mode
NODE_ENV=production

# Logging
LOG_LEVEL=info

# Request body limits in bytes
MAX_BODY_BYTES=1048576

# Comma-separated browser origins allowed by CORS in production
CORS_ALLOWED_ORIGINS=https://api.example.com,https://app.example.com

# Trusted reverse proxy IPs or IPv4 CIDR ranges
TRUSTED_PROXY_RANGES=127.0.0.1/32,10.0.0.0/8
```

## Variable reference

<ParamField query="NODE_ENV" type="string" default="(unset)">
  Runtime mode flag. When set to `production`, CORS defaults to deny-by-default unless `CORS_ALLOWED_ORIGINS` is explicitly configured. Has no effect on other behaviour.
</ParamField>

<ParamField query="LOG_LEVEL" type="string" default="info">
  Minimum log level to emit. Messages below this threshold are silently dropped.

  Accepted values (in ascending priority order):

  | Value | Description |
  |-------|-------------|
  | `debug` | Verbose developer output |
  | `info` | Standard request and lifecycle events |
  | `warn` | Non-fatal anomalies (body too large, rate limit hits) |
  | `error` | Unhandled errors and failures |
  | `success` | Successful route completions |

  Invalid or empty values fall back to `info`.
</ParamField>

<ParamField query="MAX_BODY_BYTES" type="number" default="1048576">
  Maximum request body size in bytes for routes that use `defaultMaxBodyBytes()`. The default is 1 MiB (1048576 bytes). Invalid or non-positive values fall back to the default.

  Routes can override this with a literal value in their own `createRouteBodyLimit` call — for example, a file upload route might use a higher ceiling while small JSON endpoints might use a lower one.
</ParamField>

<ParamField query="CORS_ALLOWED_ORIGINS" type="string" default="(deny all in production)">
  Comma-separated list of browser origins permitted to make cross-origin requests. Each value must be a full origin including scheme (e.g. `https://app.example.com`).

  Example:
  ```
  CORS_ALLOWED_ORIGINS=https://api.example.com,https://app.example.com
  ```

  When unset in production (`NODE_ENV=production`), all cross-origin browser requests are denied by default.
</ParamField>

<ParamField query="TRUSTED_PROXY_RANGES" type="string" default="(unset)">
  Comma-separated list of trusted reverse proxy IPv4 CIDRs or exact addresses. When the direct peer IP of an incoming connection matches this list, `X-Forwarded-For` and `X-Real-IP` headers are trusted for client IP resolution in logs and rate limiting.

  Example:
  ```
  TRUSTED_PROXY_RANGES=127.0.0.1/32,10.0.0.0/8
  ```

  When set, `TRUSTED_PROXY_IPS` is ignored.
</ParamField>

<ParamField query="TRUSTED_PROXY_IPS" type="string" default="(unset)">
  Backward-compatible alternative to `TRUSTED_PROXY_RANGES` for specifying exact trusted proxy IP addresses (comma-separated). Ignored when `TRUSTED_PROXY_RANGES` is set. Prefer `TRUSTED_PROXY_RANGES` for new deployments.
</ParamField>

<Note>
  **`CORS_ALLOWED_ORIGINS` is a security boundary in production.** When `NODE_ENV=production` and this variable is unset, the CORS middleware denies all cross-origin browser requests by default. If you need browser clients to access the API, you must list each allowed origin explicitly. Avoid using wildcard origins (`*`) with credentialed requests.
</Note>
